OSCAL Catalog Inspector XSLT

Testing and using OSCAL, the Open Security Controls Assessment Language.

This page provides a validation service. Load your OSCAL catalog and the page shows whether and where it is invalid according to the rules of the OSCAL catalog model.

Processing is entirely on the client system and entails no data transfer. XSLT logic is maintained in the OSCAL XSLT repository, from which it can also be acquired and adapted.

About the application

The Metaschema Information Modeling Framework used by OSCAL to support its data modeling provides it with a means of defining its rules abstractly such that they can be checked and enforced by an automated process, independently and agnostically with respect to the data format(s) supported by that process, while the data format is one of XML, JSON or YAML. As an XML-based technology, XSLT (XSL Transformations) now in version 3.0 is a capable language for a robust and conformant implementation of Metaschema, starting with its XML-facing capabilities.

This page applies the rules defined in the OSCAL Catalog Metaschema to XML instances, to determine their conformance to these rules, and reporting discrepancies. It does this independently of other schema technologies including the published OSCAL schemas (XSD and JSON Schema), reporting the same findings for the same inputs. Over controlled testing input, it thus offers cross-checking of technologies being examined, i.e. relevant findings for conformance testing.

As a demonstration and proof of concept, this application aims to go beyond showing the delivery of the validation capability as defined by Metaschema. Additionally, it seeks to do so in a testable, traceable and verifiable way. Accordingly it is backed by conformance testing as well as testing of its functionality and performance. In doing so, it hopes to provide second implementation confirmation of Metaschema semantics, showing their viability.

The validation capabilities of Metaschema include not only modeling constraints (such as names and permissible locations of elements to be recognized), but also arbitrary, assertion-based (or query-based) constraints over the data set itself, not only its tagging. These applications can range widely from relatively relaxed and permissive definitions to very tight ones, in which not only names and places but also tagging usage, cross-linking and "informational integrity" can be defined, cross-checked and enforced when needed. OSCAL defines a number of such rules with varying complexity, making it a good real-world use case for Metaschema.
